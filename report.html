Project report Project goals

The objective of the project is to offer a visualisation of the populations of the different types of penguins living in Antarctica.

Data Source

The main dataset used is the list of all penguin observations in Antarctica (from various sources) compiled by the Mapping Application for Penguin Populations and Projected Dynamics (MAPPPD). For each observation in the dataset, the following data is given:

  • The name given to the observation site, as well as the site initials
  • The coordinates of the site
  • The common name given to the species of penguin
  • The number of penguins observed
  • The accuracy level of the observation (1 being the most accurate, 5 the least)
  • The exact date of the observation
  • The year of the beginning of the season (For example, January 2010 observations belong to a season starting in 2009)
  • The type of penguin observed (adults, chicks, nests)
  • The source of the observation
Optionally, we are are providing a visualization for the antarctic facilities (COMNAP). Which can be freely used for educational purpose.
  • The facility names given to the installed, as well as its unique identifier
  • The facility coordinates
  • The date of installation
  • The facility type (stations, camps, refuge, airfield camp, laboratory, depot)
  • The countries owning the facilities
  • A link to the facility photos

Features

Our project is set up as an interactive website, allowing the user to discover the dataset and its implications. Using a menu at the top of each page, the user can browse different sections of the site.

  • The index of the site is a homepage, allowing access to all the pages.
  • The map page contains our interactive map, allowing the user to see the locations and species of all observed penguins for any given years, as well as a heat map of penguin activity.
  • The penguin page displays a chart of all observations, grouped by month, allowing the user to see how the data is distributed as well as see, for any given parameters, the observations with the most penguins corresponding to those parameters.
  • The final page is the one you're reading right now!

Technologies

  • The website is based on a free to use Bootstrap template from StartBootstrap. Using a base template, we can quickly start to add our own content without worrying about the basics.
  • The graphs were created using D3.js, dc.js and Crossfilter.
  • Using Crossfilter, we can select what data dimensions to consider and natively feed to DC.js graphs, which in turn uses D3.js to render and display them. Crossfilter allows interactive data filtering without completely recreating all graphs, allowing for fast modification of the data displayed.
  • Regarding the interactive map, we are using the Leaflet library and its polar projection and webgl heatmapplugins.

Usability

Here are the rules of usability according to Eric Steven Raymond and Rob W. Landley and whether the interactive map is compliant to those rules or not.

  • Rule of Bliss : The user can understand the map without prior knowledge of the UI. The time slider and filters are self-explanatory, and advanced settings are hidden by default. However, a few minutes of exploration are necessary to understand all the different types of available information (population, sites, nests).
  • Rule of Distractions : The UI doesn't require sustained attention from the user.
  • Rule of Flow : The UI doesn't disrupt the user's focus with pop-ups or anything.
  • Rule of Documentation : The interface needs a bit of an introduction to explain the meaning of the underlying data, which is an infraction to the rule of documentation.
  • Rule of Least Surprise : The UI doesn't try to impress with anything exotic or novel.
  • Rule of Transparency : The user doesn't need to memorize the state of the application unless he hides the filters.
  • Rule of Modelessness : The UI doesn't react to inputs differently depending on a hidden mode.
  • Rule of Seven : The UI shouldn't have more than seven control elements on the screen, or the user might forget their function. Unfortunately, it does.
  • Rule of Reversibility : The UI doesn't have any destructive operation. As such, it doesn't require an "undo" button. Each input can be cancelled by repeating it once.
  • Rule of Confirmation : The UI doesn't use confirmation prompts.
  • Rule of Failure : Failures should be informative. Our UI doesn't fail.
  • Rule of Silence : When nothing surprising happens, the UI should stay silent. Our map doesn't even have sound at all.
  • Rule of Automation : Automate the input of information that can be deduced. There's not much to automate on our UI but the autocompletion of the site names (which is implemented).
  • Rule of Defaults : Choose safe defaults. The default filters hide data such as human facilities, which would introduce too much visual noise on a first visit.
  • Rule of Respect : The UI should be as simple as possible. It's not, as some filters are rarely useful ("record types").
  • Rule of Predictability : Again, the UI doesn't do anything surprising, and the user instinctively knows he can click penguins to show more info, check filters to activate them, select a year with the slider, and zoom and pan with the mouse and scroll wheel.
  • Rule of Reality : User testing is important, be we were out of time to test it on a good sample of potential users.

Based on

With the goal to accelerate the development process and data analysis, we based our project on the following:

  • The interactive graphs: dc.js Getting started guide
  • The interactive map: Antarctica Stations Map

Conclusion

Unfortunately, the data isn't complete enough for us to be able to make assumptions. We actually observe an increase in the number of penguins in time, which is certainly just an increase in the frequency of expeditions rather than population. Indeed, there are more records being made in the last few years. The visualisation we provide still has the benefit of showing the wildlife hotspots and the location of human settlements.
